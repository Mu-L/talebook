Codex 纯问答与代码发布解耦方案

让已授权的 Issue 评论始终进入 Codex；只有在确实产生仓库改动时，才用分支状态决定能否发布。

状态：ACTIVE 创建日期：2026-07-18 所属模块：project 影响范围：GitHub Actions / Codex 提示词 / 工作流测试

需求来源：GitHub Issue #875 第二次提及 Codex 的失败回复；前置方案为 design/project/20260718-codex-app-comment-identity.active.html 。

原始诉求

Issue #875 中第二次评论仅询问“你的模型详细版本”，没有要求提交代码。工作流却在 Codex 执行前发现历史分支关联已关闭或已合并的 Pull Request，于是直接回复分支冲突。期望行为是：纯问答正常由 Codex App 身份回复；只有需要提交代码时才处理分支冲突或发布限制。

已复现：评论没有进入 Codex，Resolve request context 先把历史分支判为不支持目标，随后 Explain unsupported target 发布固定错误文本。

根因与设计约束

  1. supportedTarget 同时表达“能否执行请求”和“能否把代码发布到某条分支”，两个不同阶段被错误地合并。
  2. Issue 的旧 Codex 分支只要关联过已关闭或已合并 PR，就会被当作当前请求的硬错误；实际上它应当是历史记录。
  3. 现有结果协议只描述可发布代码，纯问答只能伪造 feature、提交信息和测试记录，最终回复也会带上不自然的发布提示。
  4. 不能依赖“修复、提交、为什么”等关键词判断用户意图；同一句自然语言可能既需要解释，也可能在执行后才发现需要改代码。
已确认的原则：所有通过权限校验的提及都执行 Codex；执行后根据真实 diff 和 Codex 的交付类型决定是否进入发布门禁。

目标与非目标

类型内容
目标纯问答在存在历史分支或多个活动分支时仍能正常执行和回复。
目标已关闭或已合并 PR 对应的 Issue 分支视为历史分支，不参与当前目标选择。
目标只有实际产生代码改动时才应用发布阻断，并确保阻断前不会创建 PR 或写分支。
目标结果协议明确区分 reply 与 publish，纯问答回复不显示伪造的测试或提交元数据。
非目标不通过关键词或模型调用前的分类器猜测评论意图。
非目标不改变 PR 评论对 fork、关闭 PR、缺失 head repository 的现有安全限制。
非目标不删除历史远端分支，也不自动重开或覆盖已关闭 PR。

方案

1. 将执行目标和发布限制拆开

Issue 评论的上下文解析始终给出可 checkout 的安全目标，并新增 publishBlockReason。它只是一条延迟到 diff 产生后的发布限制，不再让 supportedTarget 变为 false。

Issue 分支状态checkout 目标发布限制
没有活动分支默认分支无
恰好一个活动分支该分支无；沿用现有 PR 或恢复缺失 PR
多个活动分支默认分支若产生改动则拒绝发布，并列出分支；无改动则正常回复
仅有已关闭或已合并 PR 对应分支默认分支无；这些分支记为历史，不参与选择。若新任务生成同名发布分支，则追加本次 GitHub run ID，避免覆盖历史分支。

“活动分支”包括关联 open PR 的分支，以及从未关联 PR 的孤立分支。若同一分支同时存在历史 PR 和 open PR，以 open PR 为准。

上下文同时把现有 Issue 分支名作为 JSON 数组传给发布门禁。首次发布根据 feature 生成基础分支名；若该名称已被历史分支占用，门禁生成 基础分支名-run-id。重跑时该分支会成为唯一活动分支并被继续使用，因此不会覆盖历史提交，也不会为同一执行反复派生名称。

2. 增加显式交付类型

.codex-result.json 用枚举 delivery 替代含义模糊的 ready_to_publish：

delivery约束最终行为
reply任务已完成但不需要仓库改动；feature 和 commit_message 为空，tests 可以为空，暂存 diff 必须为空。只发布 summary；存在真实验证记录时才附加验证段落。
publish任务需要发布仓库改动；feature、Conventional Commit 和至少一条测试记录继续采用现有校验。通过 diff、设计、测试、分支和 PR 门禁后提交并发布。

若 Codex 计划发布但无法完成，仍输出 publish，通过失败测试或缺失改动由现有 fail-closed 门禁拒绝；不允许用 reply 掩盖已产生的改动。

3. 发布门禁后移到真实 diff 之后

  • reply 且无 diff：忽略 publishBlockReason，正常回复并成功结束。
  • reply 但有 diff：拒绝，防止代码改动绕过发布门禁。
  • publish 且有 diff：先应用 publishBlockReason，再执行现有设计、测试、分支和 PR 门禁。
  • publish 但无 diff：沿用“未产生仓库改动”的失败说明，不伪装成纯问答。

4. 限制所有写操作

GitHub App token 获取、缺失 PR 恢复、新建 Issue PR、提交和 push 的条件全部增加 delivery == publish。因此纯问答只有评论写入，不会意外恢复旧 PR、创建新分支或推送提交。

5. 提示词与最终回复

  • 提示词明确允许信息查询、解释和状态回答，不要求为纯问答创建不相关的设计、测试或仓库改动。
  • 提示词仍要求先实际检查仓库和上下文，再选择 reply 或 publish，而不是根据关键词预分类。
  • 纯问答最终评论仅使用经过协议校验的 summary；tests 非空时才显示“验证”，不追加“未发布”“无需提交仓库改动”等机械文本。
  • 代码交付继续保留现有 PR 链接、验证摘要和失败原因。

兼容性与安全性

  • 身份不变：进度、最终回复和 PR 仍由专用 GitHub App token 发布，继承前置方案。
  • 权限不变：未授权用户仍在 Codex 执行前退出。
  • PR 评论不放宽：关闭 PR、fork PR 和缺失 head repository 仍是执行前硬阻断，因为这些目标无法安全 checkout。
  • Issue 评论更安全：多个活动分支时只 checkout 默认分支，避免在不明确分支上修改；若无改动则可以回答，有改动则拒绝发布。
  • 结果协议为仓库内部工作流接口，无外部 API、数据库或产品数据迁移。

残余风险与替代验证

  • 本地全量 Python 环境缺少 Calibre、QuickJS 与 tomllib；已改用仓库标准 make test Docker 入口完成 627 项通过验证，覆盖后端全量回归。
  • 标准 Docker 测试镜像不含 jq，因此真实执行发布门禁的行为测试在容器内跳过；同一测试已在本机具备 bash、git、jq 的环境中通过，另由 actionlint 验证工作流语法。残余风险是本机与 GitHub Ubuntu runner 的工具版本差异。
  • GitHub API 的真实分支、PR 状态和评论更新无法在合并前端到端模拟；静态契约测试覆盖上下文路由，合并后仍需在 Issue #875 执行一次纯问答验收。若异常，可回滚本工作流提交，不影响仓库业务数据。

审查结论

  • 使用显式 delivery: reply | publish 结果协议，不做关键词分类。
  • 已关闭或已合并 PR 对应分支自动视为历史分支。
  • 多个活动分支时允许纯问答，但实际代码改动仍拒绝发布。
  • 纯问答只显示自然语言 summary，不显示空的测试和发布说明。
