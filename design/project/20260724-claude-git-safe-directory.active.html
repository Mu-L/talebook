INCIDENT FOLLOW-UP / GIT TRUST BOUNDARY

让 Git 的信任配置活过 Action 之间的 HOME 切换

问题不在 checkout 是否配置过安全目录，而在配置写进了谁的 HOME。修复必须让 Claude Action 后续执行的 Git 进程看到同一个精确工作区信任项。

创建日期 2026-07-24 所属模块 project 状态 ACTIVE 影响范围 GitHub Actions、workflow 契约测试 需求来源 Claude issue_comment 真实运行失败

1. 原始诉求与已确认事实

用户反馈合并 PR #904 后 Claude 仍执行失败，并明确要求本地完成完整测试。权威证据来自 run 30064264584 / job 89392016289 。

日志结论： unzip、Bun 1.3.14、140 个 Action 依赖、OIDC token 与 GitHub App token 均成功；失败点已经推进到 Claude Action 创建分支时执行 git fetch origin master --depth=1。 Git 返回 detected dubious ownership in repository at '/__w/talebook/talebook'。

2. 根因：配置存在，但生命周期错误

CHECKOUT / TEMPORARY HOME

安全目录写入一次性配置

actions/checkout@v4 临时覆盖 HOME，将 safe.directory 写入 runner 临时目录，以便 checkout 自己完成 Git 操作。

→HOME RESET
CLAUDE / PERSISTENT HOME

后续 Git 看不到信任项

Claude Action 恢复 job container 的持久 HOME /github/home，随后 git fetch 使用另一份全局配置，触发所有权保护。

安全边界：不得使用 safe.directory '*' 绕过 Git 所有权保护；只允许信任 GitHub 提供的当前 GITHUB_WORKSPACE 精确路径。

3. 决策与范围

采用：workflow 显式持久化

checkout 后、任何项目命令和 Claude Action 前，新增独立步骤，把当前工作区加入持久 HOME 的全局 Git 配置并立即验证。

不采用：信任所有目录

通配符能让错误消失，但扩大容器内所有仓库的信任范围，掩盖未来挂载或所有权异常。

本次不耦合：运行身份重构

root/non-root、PUID/PGID 与 TLS 权限属于另一套运行身份设计；本修复只解决 Action 间 Git 配置生命周期。

计划步骤

git config --global --add safe.directory "$GITHUB_WORKSPACE"
git config --global --get-all safe.directory | grep -Fqx "$GITHUB_WORKSPACE"
git status --short
  • 新增步骤位于 Checkout repository 之后、Verify development environment 之前。
  • git status 是行为探针：如果当前 HOME 中没有精确信任项，步骤在进入依赖安装和模型调用前直接失败。
  • 不修改 dev 镜像、不改变 job 的 root 运行方式、不扩大 GitHub permissions、不接触 OAuth secret。

5. 兼容性、风险与回滚

  • 兼容性：配置使用 GitHub 标准环境变量，不写死 runner 路径或操作系统临时目录。
  • 安全：信任范围仅为本次 job 的工作区；不使用通配符，不改变 token 或仓库权限。
  • 重复执行：--add 可能留下重复的同值条目，但 job container 一次性销毁，不跨 run 累积；验证读取所有值并匹配精确路径。
  • 外部边界：本地 act 无法等价提供 GitHub OIDC 和真实回帖/推送能力，因此线上 issue_comment 是最终闭环，不以本地 dummy token 冒充。
  • 回滚：删除新增 Git 配置步骤即可恢复原流程；不涉及镜像回滚或数据迁移。
