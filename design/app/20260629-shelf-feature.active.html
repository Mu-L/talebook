书架功能说明

将“待读”对外改名为“我的书架”，当前保持同一数据语义，不新增模型

状态 · ACTIVE关联 · PR #837迁移 · 2026-07-13
TL;DR
本次是产品命名和接口表面的统一，不是新领域模型。前端增加“我的书架”页面和入口，API 统一使用 shelf 命名，后端继续复用 ReadingState.wants。只有产品确认“书架”和“待读”需要同时存在时，才新增独立字段或模型并迁移数据。

01原始诉求

PR #837 将原有“待读”入口调整为“我的书架”。需要完成页面、导航和 API 命名的一致切换，并明确当前是否需要新增独立数据模型。

02目标

  • 用户界面统一展示“我的书架”。
  • 对外接口统一为 shelf 语义，停止注册 wants 与 case 旧接口。
  • 本次按纯改名处理，复用已有 ReadingState 数据，避免无意义迁移。
  • 记录未来产品语义分叉时的数据模型升级条件。

03方案

层级实现
页面新增 /user/shelf，登录用户导航展示“我的书架”。
新 API/api/shelf 与 /api/book/:id/shelf。
后端状态继续使用 ReadingState.wants、set_wants() 与 wants_date。
移除 API/api/wants、/api/book/:id/wants、/api/case、/api/book/:id/case 不再注册。
长期兼容性
若产品以后确认“书架”和“待读”是两个可同时存在的状态，必须新增独立字段或模型，并迁移当前复用 wants 的接口实现；不能继续依靠名称区分同一字段。
