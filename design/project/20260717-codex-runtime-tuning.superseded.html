Codex 运行时预算调优

基于 PR #871 的真实超时数据，为复杂实现任务增加执行预算，并降低默认推理强度；保留发布收尾与失败恢复空间。

状态 · SUPERSEDED 创建日期 · 2026-07-17 所属模块 · project 需求来源 · PR #871 超时复盘 影响范围 · GitHub Actions / Codex CLI / tests / design 工作分支 · fix/20260717-codex-runtime-tuning
原始诉求

把 Codex 子进程限制从 15 分钟提高到 20 分钟，将默认推理强度从 xhigh 调整为 high，并评估是否启用 fast。经用户确认，不启用 fast；Actions job 上限同步从 20 分钟提高到 25 分钟，为初始化、发布门禁、补丁上传和最终评论预留时间。

01 · 已确认事实与决策

议题决策依据与边界
Codex 执行窗口15m → 20mPR #871 中 Codex 持续工作满 15 分钟，在验证阶段被退出码 124 终止；最后仅剩约 1 分钟用于验证与结果元数据。
job 总窗口20 → 25 minutes不能让子进程与 job 同时在 20 分钟截止；额外 5 分钟覆盖约 30 秒初始化及失败恢复、发布与评论收尾。
默认推理强度xhigh → high降低复杂任务中推理等待；保留 CODEX_REASONING_EFFORT repository variable 覆盖能力。当前仓库未设置同名 variable，因此新默认会实际生效。
Fast mode不启用Codex CLI 0.144.4 已将 fast_mode 标为 stable，当前账号与 gpt-5.6-sol 真实 smoke 成功；但官方说明约 1.5 倍速度对应 GPT-5.6 约 2.5 倍 credits，用户决定保持 Standard tier。
安全与发布保持不变sandbox、凭据隔离、publish gate、App Token、fast-forward push、WIP 门禁和失败补丁均不调整。
验收口径workflow 结构必须明确呈现 25 分钟 job、20 分钟 Codex、20 分钟中文超时原因和 high fallback；不得出现 service_tier = "fast"。

03 · 实现与兼容性

  • 修改 .github/workflows/codex.yml：job timeout-minutes 改为 25；GNU timeout 改为 20m；门禁中文原因同步为 20 分钟。
  • CODEX_REASONING_EFFORT 的 fallback 改为 high，仍写入现有 model_reasoning_effort，不改变变量名和运营覆盖方式。
  • 不添加 service_tier 或 fast 配置，不修改模型 gpt-5.6-sol、CLI 版本、认证或计费路径。
  • 更新结构化 workflow 契约测试，精确锁定三个新默认值和 Standard tier 不变量。
  • 激活本方案时，将 design/project/20260716-codex-review-hardening.active.html 转为 SUPERSEDED 并链接本方案；历史文档不回写核心决策。
兼容性与风险单次失败任务最长占用 runner 的时间增加约 5 分钟；同一 PR 的后续请求仍由 cancel-in-progress: false 串行排队。20 分钟只提高上限，不保证超大任务完成；任务范围控制仍由 Prompt 与维护者请求承担。
