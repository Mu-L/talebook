Codex 数字员工工作流

结论摘要

本方案已由 design/project/20260716-codex-digital-employee-review-fixes.active.html 替代。替代方案修复了 loopback 监听、无变化门禁顺序和结构化测试评论，并保留本方案作为历史决策。

停用官方 chatgpt-codex-connector 后，仓库内 .github/workflows/codex.yml 成为唯一入口。Codex 在 20 分钟上限内获得公共出站网络、本机测试回环地址和目标分支工作区，但 permission profile 默认拒绝读取工作区外文件，因此无法读取 runner 临时目录中的登录文件、GitHub 发布令牌或仓库 Secrets；发布由后置受控步骤使用专用 GitHub App 短期 token 完成。成功任务直接推送，失败任务评论原因并保留 patch artifact。

01 · 原始诉求与现状

维护者希望在 issue 或 PR 中评论 @codex 后，机器人像数字员工一样完成修改、联网查询或安装依赖、执行验证，并把代码真正更新到对应分支。现有实现固定 checkout master、关闭 agent 网络、隐藏 Git 凭据，并且只上传 diff artifact，因此无法满足交付闭环。

此前官方 Connector 与仓库自建 Actions 同时监听 @codex，造成重复回复；官方回复中的本地 commit 也没有实际移动远端 PR head。本次以自建 Actions 为唯一自动化实现，官方 Connector 的卸载属于仓库外管理操作。

02 · 目标与非目标

类别内容
目标同仓库 PR 直接追加一个可追溯提交；普通 issue 创建或复用长期 Codex 分支及 Draft PR。
目标允许 agent 使用完整出站网络完成文档查询、依赖安装和测试，同时隔离发布 token 与 Secrets。
目标用权限校验、串行队列、fast-forward、发布门禁和 20 分钟超时限制自动化风险。
非目标不向 fork PR 推送，不允许 force-push，不允许 agent 修改 .github/workflows/。
非目标不让 Codex 自行调用带写权限的 gh，不自动合并 PR，不自动无限重试。

04 · 触发、分支与生命周期

上下文检出与交付异常处理
同仓库 PR按 API 返回的 head SHA 检出 PR head；成功后向原 head ref fast-forward push。head 在运行中移动或 push 非 fast-forward 时停止并评论；不 rebase、不强推。
fork PR不运行可写任务，不跨仓库推送。直接评论说明不支持自动更新 fork 分支。
普通 issue，首次从默认分支开始；由 Codex 生成英文 kebab-case feature，发布到 codex/issue-<id>-<feature> 并创建 Draft PR。feature 必须通过严格格式校验；否则不发布。
普通 issue，后续查找唯一的 codex/issue-<id>-* 远端分支并继续提交到已有 Draft PR。零个匹配视为首次；多个匹配、历史 PR 已关闭但分支仍存在时要求人工处理。

同一 PR 或 issue 使用 concurrency group 串行排队，cancel-in-progress: false；不同编号可以并行。Bot 评论不会再次触发任务。若首次运行已推送 issue 分支、但 Draft PR 创建失败，下一次调用在结构化契约有效且没有新 diff 时会补建缺失的 Draft PR；补建失败仍以失败状态结束。

05 · Agent 输出契约与发布门禁

Prompt 要求 Codex 在普通最终回复之外写入工作区根目录的 .codex-result.json。workflow 通过 .git/info/exclude 排除该运行产物，并将文件视为不可信输出，只接受固定字段、类型、长度和枚举值。结果文件不放入 .codex/，因为内置 workspace profile 会保护该目录为只读。

{
  "ready_to_publish": true,
  "feature": "sidebar-submenu-indent",
  "commit_message": "fix(app): 调整二级菜单缩进",
  "summary": "已将管理二级菜单缩进调整为 36px。",
  "tests": [
    {"command": "cd app && npm test -- AppHeader", "result": "passed"}
  ]
}
门禁规则
执行状态codex exec 必须成功，且 ready_to_publish 必须为 true。
测试与方案相关测试通过或合理记录未执行原因；仓库内不存在 .wip.html。
保护路径变更集中不得出现 .github/workflows/。
Git 完整性必须有真实文件变化，git diff --check 通过；忽略 .codex/tmp 运行产物。
发布动作每次调用最多一个 Conventional Commit；Bot 为 author/committer；只执行 fast-forward push。
失败交付任何门禁失败、超时或冲突都不推送。workflow 回复清晰原因，并把当前修改保存为 7 天 patch artifact，便于维护者恢复；成功路径不上传 patch。

06 · 网络、超时与命令边界

  • 使用 Codex 0.144.4 permission profile：继承 :workspace，再以 :root = deny 默认拒绝读取整个宿主文件系统，仅给 :minimal 运行时 read，并给 :workspace_roots write；其中 .github/workflows 额外降为 read-only，发布门禁再做二次检查。
  • 拒绝 :tmpdir 和 :slash_tmp；为工具设置工作区内的专用 TMPDIR。由 workflow 写入 runner 临时目录的 CODEX_HOME、认证文件和请求上下文不可被 agent shell 读取。
  • 显式启用 Codex network proxy；permission profile 开启网络并允许公共域名全局规则，同时显式保持 allow_local_binding = false，仅额外允许字面量 localhost 与 127.0.0.1 供本机渲染和服务测试使用；不开放 Docker 等 Unix socket，也不放开其他私网地址。
  • GITHUB_TOKEN、GH_TOKEN、CODEX_AUTH_JSON、CODEX_HOME 和常见 token/secret/key 环境变量继续从 agent shell 排除。
  • 依赖安装必须写入工作区；需要 Docker socket 或宿主机额外写目录的测试记录为未执行并说明原因。
  • job 的 timeout-minutes 为 20，codex exec 子进程限制为 15 分钟，为失败评论、恢复 patch 和发布收尾预留时间；超时不提交、不重试。
  • 验证过程中启动的临时服务应在任务结束前关闭，不把长期进程作为交付结果。
接受的风险公共出站网络意味着仓库内容和测试数据可能被任务命令发送到外部服务。通过仅允许 write/maintain/admin 触发、默认拒绝工作区外读取、不给 agent 任何 GitHub 写 token 或 Secrets、拒绝 fork PR、限制 20 分钟来降低风险，但无法把联网任务视为零外泄环境。

07 · 实施步骤

扩展上下文解析：记录 issue 标题、默认分支、head ref/SHA、同仓库判定和 issue 分支匹配状态。
按 PR head 或 issue 分支检出，继续关闭 checkout 凭据持久化；fork PR 在 agent 前短路。
开启 agent 网络，更新 prompt 和结构化结果契约，保持 Secrets 与 GitHub token 隔离。
实现门禁校验、保护路径检查、失败 artifact、GitHub App 短期 token、单提交和 fast-forward push。
普通 issue 首次成功后创建 Draft PR；所有上下文回写包含提交 SHA、分支、PR 和实际测试结果的评论。
新增 workflow 静态回归测试，完成本地校验后激活本方案，并将旧 Codex Actions 方案标记 SUPERSEDED。

08 · 发布、停用官方 Connector 与回滚

  1. 先合并 workflow、prompt、测试和本方案；配置并安装仓库专用 GitHub App。
  2. 用测试 issue 验证分支创建、Draft PR、CI 触发和评论；再用同仓库测试 PR 验证追加提交。
  3. 确认自建流程可用后，在 OpenAI/Codex GitHub 集成或 GitHub Installed Apps 中停用官方 chatgpt-codex-connector，避免双回复。
  4. 回滚时禁用 Codex workflow 或撤销 GitHub App 安装；删除/轮换 App 私钥即可立即切断发布能力。
  5. 若要恢复 patch-only 行为，需以新 WIP 方案替代本方案，不直接改写 ACTIVE 核心决策。
