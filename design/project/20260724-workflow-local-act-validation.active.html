WORKFLOW EXECUTION CONTRACT

Workflow 修改必须经过真实 act 本地执行

静态 YAML 正确只说明文件可解析；只有加载仓库真实 workflow、匹配真实事件并执行目标 job，才能证明第三方 Action bootstrap、容器依赖和步骤连接可工作。

创建日期 2026-07-24 所属模块 project 状态 ACTIVE 需求来源 用户要求 Claude 报错修复提交 PR 前补充 workflow 目录规范

1. 原始诉求与目标

在 .github/workflows/AGENTS.md 中明确：修改任何 GitHub Actions workflow 时，必须在本地使用真实 act 执行器测试，确保本地可复现的功能链路跑通。团队若使用 gh act 扩展入口，也必须落到同一真实执行器和真实 workflow，不能用静态检查或简化 smoke 代替。

静态校验轨 YAML 解析、actionlint、契约单测。保留，但不能证明 Action 下载、容器工具和 composite action 能运行。
→
真实执行轨 act 加载被修改的真实 workflow、匹配触发事件、运行目标 job，并记录实际越过的步骤边界。

2. 目录规范

规则明确要求
真实文件使用 -W .github/workflows/<name>.yml 加载实际提交文件；不得复制成临时简化 workflow。
真实事件使用与 workflow 触发器匹配的事件类型和最小事件 JSON，例如 issue_comment、pull_request 或 workflow_dispatch。
真实依赖job 使用本仓库构建镜像时，先构建本地 target，并以禁止强制拉取的方式确保测试实际使用本地镜像。
安全边界临时事件放在当前操作系统提供的临时路径中并在测试后删除；不得写入真实密钥。GitHub OIDC、回帖或模型调用等外部边界必须如实标注。
证据记录PR 列出 act 版本、实际执行命令、目标 job、通过步骤、预期外部边界和最终结果。
失败即阻断本地可执行步骤没有跑通，不得声明 workflow 验证通过。环境确实无法执行时，必须写明未执行原因和风险，并将其作为合并阻断项处理。

不接受：只运行 actionlint、只解析 YAML、只跑读取 workflow 文本的单元测试，或创建没有执行真实第三方 Action 的替代 smoke workflow。

3. 实现范围

  • 新增 .github/workflows/AGENTS.md ，仅约束该目录下 workflow 的修改与验收。
  • 扩展 tests/test_project_instructions.py ，锁定真实 workflow、匹配事件、禁止简化 smoke、验证记录和失败说明等关键条款。
  • 不修改现有 workflow 文件，不引入新的 GitHub Action，不改变 CI 权限。
  • 本规范与本 PR 的 unzip 修复相互印证：本 PR 已用真实 .github/workflows/claude.yml 运行到 Bun 解压、依赖安装和 OIDC 外部边界。
