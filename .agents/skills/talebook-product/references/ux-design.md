# Talebook UI 与交互设计

## 获取页面证据

评审现有体验时，只要环境可运行，就打开真实页面，不要只读 Vue 代码：

1. 记录路由、视口尺寸、浅色/深色或内置主题、登录角色和数据状态。
2. 覆盖桌面与移动端，以及正常、加载、空白、失败和权限状态。
3. 截图或准确描述可观察结果，并把判断链接到页面、组件、API 或 Issue。
4. 若环境无法启动，说明限制；把代码推断与真实渲染证据分开。

概念设计没有可运行页面时，可以使用流程或线框，但必须标注尚未验证的假设。

## 设计任务流

- 明确入口、角色、权限、前置条件、目标和成功出口。
- 先画主路径、返回路径和失败恢复，再设计页面结构。
- 检查中断、恢复、取消、重试、刷新、深链和多标签页。
- 长任务展示阶段、进度、后台继续、失败项和后续动作。
- 成功后把用户带到最可能的下一任务，避免只有“完成”提示。

## 覆盖状态

- 初始、加载、空白、无结果、部分成功、成功、失败、超时和离线。
- 未登录、权限不足、内容私藏、管理员专属和服务未启用。
- 批量操作的选择范围、部分成功、重复提交、防误删、撤销或回收站。
- 上传、扫描、导入、元数据、转换和推送等异步任务的可追踪性。

## 响应式与可访问性

- 检查桌面、窄屏和触屏下的导航、表格、筛选器、对话框与操作密度。
- 检查浅色、深色和内置主题的对比度、遮罩、滚动条、悬停和禁用态。
- 覆盖键盘顺序、焦点可见、语义标签、错误关联、触控目标和减少动效。
- 检查中英文长度变化；i18n 文案不得直接写入会破坏 vue-i18n 的字面量 `@` 或 `<`。

## 保持 Talebook 一致性

- 优先复用 `plugins/talebook.js` 的 `backend()`、Vuetify 组件和现有主题 Token。
- 对齐 AppHeader、侧边栏、书籍卡片/列表、详情页和管理后台的术语与状态。
- 明确书架、阅读状态、书单、私藏、设备、OPDS 和 WebDAV 的关系，避免重复入口。
- 外部阅读器只承诺协议能表达的能力，不假设其支持 Talebook Web 的全部状态。

## 输出交互规格

1. 用户与目标。
2. 入口、权限和前置条件。
3. 主流程与成功出口。
4. 替代、失败、取消、重试和恢复流程。
5. 页面层级、关键组件和操作优先级。
6. 桌面与移动端行为。
7. 文案、焦点、键盘和无障碍行为。
8. 字段、状态机、接口、权限和幂等要求。
9. 可观察的验收条件。
10. 非目标、依赖、迁移、风险和未验证假设。
