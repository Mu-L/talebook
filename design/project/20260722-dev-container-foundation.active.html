1. 原始诉求与最新决策

总体目标仍是让 Claude 与 Codex 最终运行在 Talebook 的统一开发镜像中，并能够完成后端和前端开发测试。但交付必须拆分：第一个 PR 只建立、发布并验证 talebook/talebook:dev；镜像上线后，后续 PR 才修改 claude.yml 与 codex.yml。

明确决策：agent-dev-smoke.yml 不是线上长期使用的 workflow，不得提交到仓库。需要的本地 GitHub Actions 验收使用 /private/tmp 下的一次性 workflow，测试完成后不进入 git。

2. 为什么拆分

若同一个合并同时发布镜像并让 Claude/Codex 立即引用 :dev，消费者可能在 master 的镜像构建完成前触发，从而拉取旧镜像或因标签不存在而失败。拆分后的发布顺序把镜像可用性变成第二个 PR 的明确前置条件。

4. 实现方案

  1. 调整 Dockerfile：让 dev 继承 test，安装 agent 与开发工具，并在构建时执行版本探测。
  2. 调整 .github/workflows/build.yml：复用现有 build job，在 PR 中只验证构建，在 master push 中发布唯一的 :dev 标签，并写入 OCI revision。
  3. 添加 Docker stage 与 build workflow 契约测试，锁定继承关系、工具、发布条件和标签策略。
  4. 本地构建同名镜像后，动态生成临时 workflow 并用 act -W /private/tmp/... 验证 job container 能执行工具探测和开发测试；临时文件不复制进仓库。

5. 发布、失败与回滚

  • PR 构建失败时不能合并，因此不会影响 Docker Hub 上已有标签。
  • master 发布失败时，后续消费者 PR 保持阻塞，不修改 Claude/Codex workflow。
  • :dev 是维护者确认的浮动标签；OCI revision 与构建日志用于追溯实际来源。
  • 如新镜像不可用，在消费者尚未迁移时不会影响 agent；修复 Dockerfile 后重新由 master 发布即可。
