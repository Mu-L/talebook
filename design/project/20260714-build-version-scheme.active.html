非 tag 构建版本号方案（git describe）

创建日期：2026-07-14 所属模块：project（发布流程；影响 .github/workflows/build.yml 、 Makefile ） 状态：ACTIVE 需求来源：线上容器 update_checker 误报“已是最新”，排查发现 VERSION=52957175 为 commit 短 hash。

1. 原始诉求

用户运行的容器日志出现 update_checker:142 Already up to date: version=52957175 。 52957175 并非用户所打的 tag，而是 commit 5295717539… 的前 8 位短 hash。用户要求：

“优化一下，non-tag 的应该使用 v26.07.13-xxx 这种版本号格式。”

2. 背景与根因

镜像内 webserver/version.py 的 VERSION 由构建时 build-arg GIT_VERSION 写入 （ Dockerfile 第 143 / 265 行）。CI 中 GIT_VERSION 的取值逻辑为： 

# .github/workflows/build.yml prepare 阶段（现状）
if [[ $GITHUB_REF == refs/tags/* ]]; then
  VERSION=${GITHUB_REF#refs/tags/}     # tag 推送 → v26.07.13
else
  VERSION=${GITHUB_SHA::8}             # 分支推送 → 52957175（纯 hash）
fi

分支（master / dev/* 等）推送产出的镜像 VERSION 是纯 hash。 update_checker._version_parts() 用 re.findall(r"\d+") 抽数字，把 52957175 当成首段版本号，远大于 release tag v26.07.13 的首段 26 ， 于是判定“当前更新” → 误报“已是最新”。

注：用户确实为 5295717 打了 tag v26.07.13 。tag 推送会另出一份 VERSION=v26.07.13 的 :latest / :v26.07.13 镜像； 问题在于运行的容器用的是分支构建镜像。本方案只修“非 tag 构建的版本号格式”，让分支镜像也带上有意义、可比较的版本号。

3. 目标

  • 非 tag 构建的 VERSION 使用 git describe 语义化格式：最近 tag + 距离 + 短 hash，例如 v26.07.13-5-g5295717 。
  • 该格式与现有 update_checker 比较逻辑天然兼容，不再误报“已是最新”，且开发版领先于 release 时不误报“有更新”。
  • 本地 make build 与 CI 保持一致。
  • 改动最小：不改 update_checker 比较算法，不改 Dockerfile 写入方式。

4. 方案

4.1 CI：build.yml prepare 阶段

当前 prepare job 无 checkout，仅凭环境变量算版本，因此无法调用 git describe 。 新增一次全历史 checkout（fetch-depth: 0 ，含 tags），并把非 tag 分支改用 git describe ： 

- name: Checkout
  uses: actions/checkout@v4
  with:
    fetch-depth: 0

- name: Prepare build args and safe ref name
  id: prep
  run: |
    if [[ $GITHUB_REF == refs/tags/* ]]; then
      VERSION=${GITHUB_REF#refs/tags/}
    else
      VERSION=$(git describe --tags --always --match 'v[0-9]*')
    fi
    ...
  • --match 'v[0-9]*' 只匹配 v26.* / v3.* 等版本 tag，排除 base-v8.5.0 之类。
  • --always 兜底：仓库无匹配 tag 时退回短 hash（极端情况，不引入新分支逻辑）。
  • tag 推送分支保持不变，仍用 tag 名。

4.2 本地：Makefile

当前 VER 既作镜像 tag 又作 GIT_VERSION 。拆出独立的 GIT_VER 供 build-arg 使用，镜像 tag 仍用分支名： 

GIT_VER := $(shell git describe --tags --always --match 'v[0-9]*' 2>/dev/null || git rev-parse --short HEAD)
# build-spa / build-ssr: --build-arg GIT_VERSION=$(GIT_VER)

4.3 为什么无需改 update_checker

git describe 格式与现有比较器行为验证：

当前版本（容器内）latest release抽取数字对比结果
v26.07.13-5-g5295717 v26.07.20 [26,7,13,…] vs [26,7,20]，第三段 13<20 有更新 ✅
v26.07.13-5-g5295717 v26.07.13 [26,7,13,5,…] vs [26,7,13]，开发版领先 无更新 ✅（不骚扰）
52957175（旧） v26.07.13 [52957175] vs [26,…]，首段巨大 误报无更新 ❌

新格式首段恒为年份 26 ，彻底避开“纯 hash 被当巨型版本号”的陷阱。

4.4 取舍

  • 不额外加固 update_checker 拦截纯 hash：仅当仓库完全无 tag 时才会退回纯 hash（--always 兜底），属极端场景；按最小改动原则不引入额外分支。此为已知残留边界，非本次目标。
  • 不改 Dockerfile 版本写入方式：仅调整传入的 GIT_VERSION 取值。
