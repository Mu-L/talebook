1. 原始诉求与事实

继续保留 talebook/talebook:dev 作为 Claude 与 Codex 的统一业务环境，修复 Claude Action 在 job container 内无法安装 Bun 的问题。用户要求本地验收必须加载真实 .github/workflows/claude.yml 并模拟 @claude，不能再用只执行工具探测和项目检查的替代 smoke 冒充 Action 验收。

证据结论
线上 run 29974489717checkout、环境探测和依赖同步成功；Claude Action 内部 setup-bun 下载 Bun 后报 Unable to locate executable file: unzip。
本地真实 workflow 复现act v0.2.82 加载仓库 .github/workflows/claude.yml ，使用 issue_comment 事件与 dummy OAuth token，稳定复现同一 unzip 错误；临时事件文件已删除。
旧测试缺口旧临时 act workflow 没有执行 Claude Action；YAML contract 只检查步骤顺序和参数；PR 上的 Claude review 是另一份运行在宿主 ubuntu-latest 的 workflow。

纠偏：旧方案所称“等价 job”只等价到业务环境和项目命令，不等价到第三方 composite action 的 bootstrap 边界。本方案把真实 Action bootstrap 纳入验收。

  1. 先给 test_dev_stage_is_a_complete_agent_development_environment 增加 unzip 安装和版本探测断言，在当前 Dockerfile 上确认红灯。
  2. 最小修改 dev stage：在已有 apt 清单加入 unzip，并在工具链验证 RUN 中加入 unzip -v。
  3. 构建本地 dev target，并直接验证 unzip。
  4. 创建一次性 /private/tmp 事件 JSON，使用 dummy token 运行仓库真实 .github/workflows/claude.yml ；验收点是 Action 内部 Install Bun 成功并进入后续步骤。因 dummy token 导致的后续鉴权失败不视为 bootstrap 失败。
  5. 删除临时事件文件，确认仓库中不存在 smoke workflow。

本地 act 可以验证真实 composite action、Bun 下载和解压链路，但不能等价证明 GitHub OIDC、仓库 API 回帖和真实模型调用；这些能力由发布后的真实 @claude 验收。

4. 发布、风险与回滚

  • 发布仍由 master 的 build workflow 覆盖浮动 talebook/talebook:dev，不增加 SHA 标签。
  • 风险仅为 dev 镜像增加一个 Debian 工具包及少量体积；production 镜像不继承 dev stage。
  • 若 Bun bootstrap 仍失败，保持方案 WIP，依据真实 Action 下一条错误补充缺失依赖，不提前扩大到 Claude/Bun 预装。
  • 回滚为移除 dev stage 的 unzip 安装和探测；不会影响生产镜像。
  • 发布后真实 @claude 未成功前，不启动 Codex sandbox 修复 PR。

5. Codex 新失败证据的隔离结论

run 30006177093 使用新下载的 dev 镜像，Docker 已收到 --security-opt seccomp=unconfined；所有工具探测成功，但 codex sandbox -- /bin/true 返回非零。当前 workflow 使用 2>/dev/null 丢弃了原始错误，因此只能定位到 sandbox 初始化失败，不能从现有日志证明具体 AppArmor/user namespace 分支。

后续独立 Codex 方案必须先取消 stderr 屏蔽并记录宿主策略诊断，再决定是否增加 AppArmor container option 或调整 runner 架构。本 PR 不携带该改动。
