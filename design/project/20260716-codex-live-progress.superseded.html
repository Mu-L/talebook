需求复述

现有工作流仅在接单时添加 🚀 reaction，并在全部工作结束后新建一次结果评论。用户需要在任务处理中看到 Codex 已接单、已经形成的计划、当前执行阶段和验证/发布状态，而不必打开 Actions 日志等待最多 20 分钟；同时，Codex 的受信任提示词和面向维护者的答复需要统一使用中文。

01 · 目标、约束与已确认决策

维度决策验收标准
评论模型保留 🚀，另创建一条状态评论并持续更新；最终结果覆盖同一条评论。一次请求最多产生一条 Codex 时间线评论，不因阶段变化刷屏。
计划可见性使用 codex exec --json 消费 JSONL 中的 todo_list 计划事件。首次计划和后续完成状态无需等待任务结束即可显示。
回复语言受信任 Prompt 改为中文，并要求计划、进度文本、结构化摘要和最终答复使用中文；命令、路径、字段名和 Conventional Commit 保持既定格式。GitHub 进度及最终结果默认中文，机器契约保持兼容。
更新频率计划变化和阶段切换立即更新；普通命令、文件、搜索活动最多每 60 秒汇总一次。用户能看到持续进展，同时避免 GitHub secondary rate limit。
信息边界展示计划文本、阶段和活动计数；不展示 reasoning、原始命令、命令输出、文件内容或环境变量。评论中不泄露凭据、内部推理或不必要的执行细节。
失败降级实时更新失败不终止编码任务；最终步骤仍尝试更新原评论，找不到时退化为新建评论。GitHub API 短暂失败不丢失代码结果，最终交付仍可见。

02 · 数据流与安全边界

  • Prompt 与进度桥接器都从默认分支读取到 Runner 临时目录，不执行 PR head 中可被修改的桥接代码。
  • 桥接器转义 Markdown、打断用户 mention、限制计划项数量和长度，并忽略 reasoning 与原始工具载荷。
  • JSONL 同时保存在被 Git 排除的 .codex/tmp/，仅供最终阶段和故障诊断使用。
  • 实时更新使用现有 issues: write/pull-requests: write，不提升顶层 contents: read。

03 · 状态与失败语义

  1. 已接收：目标与权限确认后立即创建状态评论。
  2. 分析/计划：turn.started 后显示分析中；todo_list 到达后立即呈现计划。
  3. 执行：首次命令、文件修改或搜索事件触发阶段切换；计划完成状态随事件更新。
  4. 验证：turn.completed 后显示正在验证并准备发布。
  5. 完成：现有 publish gate、commit、push、Draft PR 流程结束后，用结构化结果覆盖状态评论。

桥接器异常只写入 Actions warning/stderr，不影响 Codex 管道退出码；Codex、门禁或发布失败仍由原有 fail-closed 逻辑决定最终状态。

04 · 实施与回滚

  1. 新增受信任的 Python 进度桥接器及纯逻辑测试，覆盖计划渲染、脱敏、节流信号和 GitHub PATCH。
  2. 把受信任 Prompt 改为中文，并增加中文计划、摘要和最终答复约束。
  3. 扩展上下文步骤：从默认分支取桥接器、创建初始评论并输出 comment id。
  4. 将 Codex 改为 --json，用管道把事件复制到本地并交给桥接器，同时显式移除 token。
  5. 最终评论优先更新 comment id，API 更新失败或 id 缺失时回退到创建新评论。
  6. 回滚时恢复单次最终评论与非 JSON 输出；新建替代方案，不改写生效文档。
