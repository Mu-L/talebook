原始诉求

创建 design/ 目录作为项目设计文档目录。每次重要改动应创建方案文档，采用模块、日期、功能和状态组成的路径。

方案应覆盖原始需求、架构设计、业务流程、库表设计、数据规模、长期兼容性和测试结果等与本次改动相关的内容，并使用表格、流程图、架构图等图表降低阅读成本。

开发过程分为“复述需求、grill 澄清、输出方案、编码开发、测试验证”五个阶段，测试完成后必须回写方案。

澄清后的关键约束

  • document/ 面向产品使用者，design/ 面向内部开发者。
  • 内部方案必须使用中文单文件 HTML，不要求固定模板和固定章节顺序。
  • 必备内容是原始诉求、目标、方案和测试结果。
  • 开发中的方案使用 .wip.html ，测试完成并准备合并时转为 .active.html 。
  • 自然语言输出路径时，路径与后续标点之间必须留一个空格；代码和链接语法内部除外。

目标

形成决策记忆

让维护者知道需求从哪里来、为何采用当前方案、哪些约束不能被后续重构无意破坏。

把验证写回方案

测试不再只存在于终端输出或最终回复中，而是成为当前有效设计依据的一部分。

降低阅读成本

使用离线 HTML、清晰排版和有信息价值的图表，使长方案仍然可导航、可扫描。

提供轻量门禁

机器负责客观格式和状态约束，开发者与评审负责判断方案是否必要、内容是否充分。

适用范围

必须创建方案 通常可以豁免
新功能、用户可感知行为变化 错字、注释、小型文档修正
API、数据结构、库表、权限变化 纯格式化和 lint 修复
部署、配置、兼容性和跨模块变化 不改变业务行为的测试调整
性能、安全、迁移和复杂缺陷修复 不改变行为的小型重构
工程流程与开发规范变化 其他能够明确说明无设计影响的微小改动

是否需要方案由开发者判断，但最终回复或 PR 必须给出方案路径，或明确说明豁免原因。

方案

五阶段工作流

方案文档不是编码完成后的补充材料，而是在澄清结束后、编码开始前建立，并持续更新到验证完成。

文档路径

design/<module>/yyyymmdd-<feature>.<status>.html

  • <module> 优先使用真实代码目录，项目治理统一使用 project。
  • <feature> 使用小写英文 kebab-case。
  • 日期使用方案首次创建日期，后续内容更新不改变日期。
  • 跨模块方案归入主要责任模块，并在正文列出影响范围。

生命周期与合并门禁

状态 允许的操作 禁止事项
WIP 持续更新；未生效时删除 进入主分支或合并
ACTIVE 修正错字、链接和非实质说明 删除或直接重写核心决策
SUPERSEDED 补充替代方案与替代原因 继续作为当前设计依据

自动校验

新增公开命令 make check-design，并作为独立步骤接入 CI。机器只检查客观约束，不替代设计评审。

自动检查 人工判断
目录层级、文件名、日期和状态后缀 本次改动是否需要创建方案
HTML 基本结构和中文语言声明 原始诉求、目标、方案和测试结果是否充分
远程脚本、样式、字体、媒体和 CSS 资源 图表是否有信息价值、设计是否合理
待合并内容中是否存在 WIP 文档 豁免理由是否成立

现有文档迁移

迁移作为规范落地后的独立改动执行，本次不移动现有文档。

  1. 面向使用者的安装、使用和接口说明继续保留在 document/ ，不强制改变格式。
  2. 内部技术方案和进展记录迁入对应 design/<module>/ 并转换为单文件 HTML。
  3. 尽量使用 git mv 保留历史，并在文档顶部记录原路径和迁移日期。
  4. 仍代表当前实现的文档标记为 ACTIVE；已过时的历史文档标记为 SUPERSEDED。
  5. 迁移旧文档没有明确替代方案时，允许注明“由当前实现取代”及失效原因。
