ARMv7 Python 依赖 Wheel 构建

创建日期：2026-07-18 所属模块：docker 状态：WIP 需求来源：PR #872 及其他开放 PR 的 CI 故障
在 Talebook 应用镜像中增加一次性 Python wheel 构建阶段，为缺少 ARMv7 预编译包的 psutil 与 cffi 构建 wheel；最终 server 先离线安装这两个构建产物，再按原路径安装其余生产依赖。production 与 test 均不携带 GCC、Python headers 或其他编译工具。

原始诉求与故障证据

用户要求解决 PR #872 红灯，并把最新 master 合入其他开放 PR。同步后，三个互不相关分支的 linux/arm/v7 应用镜像均在相同位置失败。日志显示 psutil 与 cffi 没有可用 ARMv7 wheel，pip 回退源码编译后找不到 arm-linux-gnueabihf-gcc。

依赖来源ARMv7 行为
psutilTalebook 直接依赖；转换服务用于识别正在运行的转换进程只取得源码包，需要本机编译扩展
cffisocial-auth-core → cryptography → cffi只取得源码包，需要本机编译扩展
bcryptTalebook 直接依赖；密码哈希与迁移已有 ARMv7 wheel，不是本次故障来源

目标、边界与决策

  • 继续发布 amd64、arm64 与 arm/v7 的 SPA/SSR 应用镜像。
  • 编译工具只存在于不可发布的中间阶段，生产镜像继续保持 slim 边界。
  • 应用依赖的构建责任留在 Talebook 仓库；talebook-base 只维护通用 Calibre 运行环境。
  • 生产和测试阶段使用同一组 psutil、cffi wheel，避免再次回退源码编译。
  • 不删除 psutil、社交登录或密码哈希能力，不改变业务行为。

方案

文件计划改动
Dockerfile抽取公共基础阶段；新增 wheel builder；仅在 ARMv7 安装工具链并构建 psutil/cffi；server 通过只读挂载先离线安装这两个 wheel。
tests/test_docker_stages.py固定 builder 工具链、wheel 交付方式和最终阶段不包含工具链的边界。
本方案记录根因、实现、真实测试与多架构 CI 结果。

风险、回滚与验收

主要风险：ARMv7 首次构建 psutil/cffi 会增加约两分钟；BuildKit 缓存应让相同 Dockerfile 与架构复用结果。wheel builder 和 server 必须基于同一基础镜像，防止 Python ABI 或系统库不一致。其余未固定版本依赖仍沿用原有在线解析行为，不在本次扩大锁定范围。

若多架构验证失败，可回滚 Dockerfile 与结构测试；不需要回滚基础镜像，也不会修改业务数据。验收以三个架构的 SPA/SSR 构建成功、生产镜像导入 psutil 与 cffi 成功、最终镜像缺少 GCC 为准。
