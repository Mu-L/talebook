TALEBOOK · BUILD DESIGN

Talebook 本地构建边界收敛

创建日期：2026-07-18 所属模块：project 状态：ACTIVE 需求来源：slim base test-server 日志复核
Makefile 的本地镜像 tag 与镜像内版本统一使用分支名，并改为只有构建目标引用时才求值；删除所有 push 目标。基础镜像源码与发布 workflow 也从 Talebook 删除，统一由独立 talebook-base 仓库维护。

原始诉求与根因

用户观察到无 Git 的 slim base 执行 make init、lint、pytest 时出现两次 git: not found，提出移除本地特殊版本编译、统一使用分支名版本，并明确 Makefile 不再提供 push。随后确认本仓库的 Dockerfile.base 也应删除，因为 slim base 已迁移到独立 talebook-base 仓库维护。

Git 日志的直接根因是 Makefile 顶层以立即展开赋值计算 VER 与 GIT_VER；GNU Make 解析任意目标时都会运行两条 Git 命令，即使目标完全不构建镜像。仓库同时保留旧 base Dockerfile、Makefile target 和发布 workflow，则会形成两个可修改、可发布但可能漂移的事实来源。

决策理由
本地只保留一个 VER镜像 tag 和 Docker build-arg GIT_VERSION 都使用分支名，符合本地构建仅用于验证的定位。
相关变量使用递归赋值仅当 build 配方展开它们时才调用 Git；测试和检查目标不再隐式依赖 Git。
删除 push 与 push-baseMakefile 只保留本地构建验证能力；Talebook 正式镜像由 GitHub Actions 发布，base 镜像由独立 talebook-base 仓库发布。
删除本仓库 base 源码与 workflowDockerfile.base 和 build-base.yml 已由独立仓库取代；消费仓库只固定使用发布 tag，避免双源维护。
不修改 .github/workflows/build.yml真实发布仍需 git describe 产生可被 update checker 正确比较的版本，避免退回纯 hash 或无语义分支版本。

方案

VER = $(shell git branch --show-current | tr '/' '-')
IMAGE = talebook/talebook:$(VER)
TAG2 = talebook/talebook:server-side-render-$(VER)

build-spa / build-ssr:
    docker build ... --build-arg GIT_VERSION=$(VER) ...

删除 GIT_VER、push、push-base 及对应 .PHONY 声明。REPO1、REPO2 和固定 SSR tag 不含动态变量，可继续立即赋值。发布 workflow 不调用这些 Makefile 目标，因此其 git describe 版本策略不受影响。

兼容性与替代关系

  • 本方案生效后，完整替代 design/project/20260714-build-version-scheme.superseded.html：保留其 CI 真实发布的 git describe 策略，替换其 Makefile 本地 GIT_VER 策略。
  • 同时替代 design/docker/20260602-calibre-docker-merge.superseded.html 与 design/docker/20260612-docker-base-image-slim.superseded.html 所描述的“base 源码由 Talebook 维护”状态；历史方案转为 SUPERSEDED，当前维护权归独立仓库。
  • 本地构建的镜像内 VERSION 从最近 tag 描述改为分支名；这是用户确认的本地验证语义，不影响 Docker Hub 发布产物。
  • 本地不再存在上传镜像的 Makefile target；需要发布时只能走受审计的仓库 workflow。
  • 若在 detached HEAD 下执行本地 build，分支名可能为空；当前本地流程以工作分支构建为前提，不为不再承担发布职责的路径增加 hash fallback。
