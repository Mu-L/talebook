网络小说“保存到本地”进度展示方案

以内存任务台账打通已有进度数据与当前页面，持久化任务中心留待后续

状态 · ACTIVE原日期 · 2026-06-07迁移 · 2026-07-13
TL;DR
online_save 已持续写入保存进度，但 BackgroundService 没有 HTTP 查询接口，前端因此不可见。本方案在不新增数据库表的前提下引入稳定 tag、状态接口与页面轮询，进程重启后的状态丢失由明确提示和站内信兜底。

01原始诉求

用户在网络书详情页点击“保存到本地”后，只能看到任务已启动，无法查看类似“123/1546 章”的实时进度。需要让当前页面展示任务执行、完成与失败状态，并在刷新页面后尽量恢复进度。

02目标

  • 复用已有 BackgroundService 进度数据，以最小范围打通 online_save。
  • 不引入新表，不改造 search、scan 等其他异步任务。
  • 页面按 1.5 秒间隔轮询，支持运行中、完成、失败和任务丢失状态。
  • 用业务 tag 定位任务，不依赖进程内自增 ID 的跨重启唯一性。

03现状与方案

机制用途存储可见性
SearchTaskService网络书库搜索内存 + TTL已有专属 status 接口，前端闭环完整
BackgroundServiceconvert、scan、audio、online_save 等通用任务纯内存有内部查询方法，无 HTTP 暴露，前端不读取
Message最终成功或失败通知数据库用户可从顶栏铃铛查看

后端任务台账

  • BackgroundTask 增加默认空字符串的 tag；add_task() 接收 tag；get_task_by_tag() 返回同 tag 最新任务。
  • NetworkSave 在请求线程同步创建任务，再启动异步保存，保证第一次轮询时任务已登记；相同 tag 已有运行中任务时直接复用，避免重复抓取整本小说。
  • save_online_book 接收 source_id 与 book_url，构造 online_save:{source_id}:{book_url}。
  • 抓章循环维持每 20 章更新一次 {total, done}；成功补写 book_id，失败调用已有的 complete_task(error_message=...)。

HTTP 与前端

  • NetworkSave 返回 tag；GET /api/network/save/status?source_id=&book_url= 返回 found、status、progress、done、total、book_id 与 error。
  • app/pages/network/book.vue 按 1.5 秒间隔轮询；完成时展示本地书链接，失败时展示错误。
  • 仅当当前浏览器会话的 sessionStorage 保存过该书的启动标记时，进入或刷新详情页才恢复查询；组件卸载时递增 token 终止旧轮询。
  • 查询失败或 found:false 连续出现 3 次后才判定任务丢失，容忍瞬时请求失败。
已知局限
后端进程重启会清空任务，浏览器会话结束也不会跨会话恢复进度。页面连续 3 次查询失败或得到 found:false 后停止轮询，并提示“无法获取进度（可能已完成或服务重启），请查看站内通知”。成功与失败均由 Message 提供最终兜底。

后续兼容性

将来可把 BackgroundService 迁移为持久化 AsyncTask 表，并沿用 tag/output 约定，提供通用任务接口与任务中心。SearchTaskService 是否合并应另行决策。

04当前实现核对

能力代码证据
任务定位与去重background_service.py 提供 get_task_by_tag()；保存接口同步登记任务并复用运行中的同 tag 任务。
进度写入save_service.py 每 20 章或结束时调用 update_progress()。
状态恢复book.vue 使用 sessionStorage 标记、1.5 秒轮询、3 次 miss 宽限和卸载 token。
持久性任务台账仍在内存中，未引入数据库表；进程重启后的实时状态不可恢复。

05测试结果

现有 tests/test_network_save.py 覆盖 txt、epub、参数错误、任务不存在和进度响应；app/test/e2e/network.spec.ts 覆盖前端从 40/100 进度到完成链接。本轮未执行这两组用例：后端接口测试需要当前环境未安装的 Calibre，前端 E2E 需要启动 mock API 与 Nuxt；本次结论来自代码与测试用例静态核对，不把未运行用例记为通过。
