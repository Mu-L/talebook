管理员访问私藏图书权限统一方案

修复管理员在“图书管理”能搜到图书、在普通搜索却看不到的问题，并统一私藏图书在各入口的访问规则。

状态：ACTIVE 创建日期：2026-07-16 所属模块：webserver 影响范围：网页 API、OPDS、WebDAV 需求来源：GitHub Issue #867

原始诉求与问题定位

需求来源：talebook/talebook#867。同一个关键词在后台“图书管理”中能搜索到图书，但顶部普通搜索无结果。用户进一步确认：管理员本身具有查看权限，应当能够查看其他用户的私藏图书。

根因：后台列表直接使用 Calibre 搜索结果；普通搜索经过 render_book_list() 后，会把所有“非当前用户所有”的 scope=private 图书过滤掉。该过滤只按所有者判断，没有为管理员豁免。

代码审计还发现，详情与元数据参考接口各自重复了同类判断，而阅读、下载和 EPUB 文件入口没有统一检查私藏所有权。结果既可能出现“搜得到但打不开”，也可能出现“列表隐藏但直接链接可访问”。

目标与非目标

目标

  • 管理员通过普通搜索可找到其他用户的私藏图书。
  • 管理员在首页、书库、榜单、详情、阅读、下载、OPDS 与 WebDAV 中具有一致的查看能力。
  • 所有者仍可访问自己的私藏图书。
  • 其他用户与游客无法通过列表或直接链接访问私藏图书。
  • 用自动化测试固定权限矩阵，避免入口之间再次漂移。

非目标

  • 不改变私藏图书的所有者与 scope 数据结构。
  • 不改变管理员和所有者现有的编辑、删除权限。
  • 不修改 Calibre 的搜索语法、分词或索引机制。
  • 不新增前端页面、开关或 API 字段。

确认后的权限矩阵

访问者公开图书自己的私藏图书他人的私藏图书
管理员允许允许允许
普通登录用户允许允许拒绝并按不存在处理
游客服从现有游客阅读/下载配置不适用拒绝并按不存在处理

“按不存在处理”延续现有隐私策略，不向无权用户泄露私藏图书是否存在。游客配置仅决定公开图书能否阅读或下载，不能越过私藏权限。

方案

1. 统一网页端权限判定

在 BaseHandler 中提供单本图书可见性判断，并让批量私藏过滤首先识别管理员。get_books() 默认移除当前访问者不可见的私藏图书；get_book() 先执行单本权限判断，再读取 Calibre 数据，避免为单本请求扫描全部私藏 ID。两者共同覆盖详情、阅读、下载、TXT/EPUB 内容、收藏/阅读状态列表、邮件发送和已有的书籍操作入口；无权访问时沿用不存在语义。已经预先过滤 ID 的分页列表显式跳过第二次权限查询，避免重复 SQL。

首页与热度榜单移除重复的私藏过滤分支，复用同一权限规则。元数据参考接口在读取 Calibre 元数据前执行统一判定。

2. 补齐不经过 get_book() 的内容入口

封面与 OPF 文件接口直接访问 Calibre 数据库，需要显式执行单本可见性判断。公开图书行为不变；默认封面不对应具体书籍，不受影响。

3. OPDS 与 WebDAV 保持一致

OPDS 已复用 BaseHandler 的批量过滤，管理员豁免会自然覆盖搜索、全部图书和分类结果。WebDAV 使用独立 provider，因此在计算不可见私藏 ID 时读取认证用户的管理员标记；管理员返回空的不可见集合，普通用户和游客保持原规则。列表和按文件名直接解析都继续复用这组 ID。

实现约束与风险控制

  • 管理员判断沿用现有 Reader.is_admin() 与 BaseHandler.is_admin()，不引入第二套角色定义。
  • 批量列表使用一次 SQL 查询取得不可见 ID，避免逐书查询。
  • 单本判断只查询对应 Item；没有 Item 或非 private 时按公开图书处理，兼容存量数据。
  • 游客的公开图书能力仍由 ALLOW_GUEST_READ、ALLOW_GUEST_DOWNLOAD 等现有配置控制。
  • 无前端代码和用户界面变化，因此无需新增 i18n 文案，也不需要前端构建产物变更。
