DESIGN / APP / STATE RECONCILIATION

书籍阅读状态刷新一致性修复

让“加入书架”和“阅读状态”在首次打开、硬刷新、切换书籍及登录态变化后，始终以服务端已持久化状态为准。

ACTIVE · 已验证 创建日期 2026-07-22 模块 app 分支 fix/20260722-book-reading-state-refresh

01. 原始诉求与现场证据

用户提供的视频显示：登录后在书籍详情页执行“加入书架”或“设为在读”，当前页面会即时变化；刷新页面后，按钮却退回默认状态，造成操作未保存的错觉。

需求来源
用户于 2026-07-22 提供的本地录屏及随后确认的修复边界。

检查项结果结论
本地浏览器基线复现操作后显示“移除书架”；硬刷新后重新显示“加入书架”问题稳定存在于 UI 初始化阶段
GET /api/book/{id}/readstate刷新后仍返回 wants: true服务端已经持久化，不能通过重复写入修复
页面状态监听仅监听书籍 ID 变化，且没有 immediate首次挂载时 ID 已就绪，监听器可能永不触发
登录态监听登录变化只刷新设备列表阅读状态不会随登录/退出同步或清空

02. 目标与边界

必须达成

服务端真值驱动 UI

  • 首次进入及硬刷新时拉取当前书籍状态。
  • 书籍路由切换、登录成功时重新同步。
  • 退出登录或没有有效书籍时清空用户态，避免串号。
  • 快速切换时丢弃过期响应，避免旧书状态覆盖新书。

明确不做

保持现有业务契约

  • 不修改后端 API、数据库或权限模型。
  • 不改变“加入/移除书架”“在读/读完”的按钮语义。
  • 不新增本地持久化或乐观状态队列。
  • 不借此重构整页数据加载架构。

03. 状态账本：默认值与服务端真值合流

UI 安全默认值

未加入 · 未标记

仅用于未登录、书籍无效或请求尚未完成时，不能覆盖已经返回的服务端状态。

isInShelf = false readingState = 0

API 权威状态

按书籍、按用户恢复

/readstate 返回后更新书架、阅读状态和最近阅读时间，并根据 read_date 推导已阅读天数；页面展示以该响应为准。

wants read_state read_date derived read days

04. 同步流程

  1. 使用同一个响应式监听器观察当前路由书籍 ID 与登录状态，并启用 immediate: true。
  2. 每次触发先恢复安全默认值；未登录或书籍无效时到此结束。
  3. 已登录时请求当前书籍的 /readstate，成功后填充所有相关字段。
  4. 提交响应前再次核对当前书籍 ID 和登录状态；不匹配说明响应已经过期，直接丢弃。
  5. 保留按钮操作成功后的即时反馈，下一次同步再由服务端确认最终状态。

05. 实现影响

验收环境说明

自动化浏览器覆盖使用的 mock 未实现 /api/book/txt/init 与 /api/user/devices，因此这两个既有请求返回 404，并触发现有错误弹窗；同时开发模式存在详情页 SSR hydration warning。它们在本次改动前已存在，不涉及 /readstate。后续已在真实 Docker dev 后端完成独立复验，真实验收不使用 mock；因原生 Nuxt watcher 在当前 worktree 触发 EMFILE，且工程约束禁止 polling，前端使用同一源码的 production preview。

07. 风险、监控与回滚

主要风险

  • SSR 与客户端监听造成重复请求：以实际网络记录确认，必要时保持幂等并限制触发源。
  • 快速切书响应乱序：用请求对应的书籍 ID 与当前身份做提交前校验。
  • 退出后残留上一用户信息：同步入口统一先重置四个用户相关字段。

回滚策略

改动集中在详情页同步逻辑及测试 mock。若线上出现回归，可整体回退该提交；由于没有 API、数据库或数据迁移变更，不需要额外回滚步骤。

可观察信号
详情页 /readstate 请求次数、响应错误提示、刷新前后按钮和阅读状态一致性。
