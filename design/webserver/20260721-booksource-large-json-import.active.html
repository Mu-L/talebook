修复大体积书源 JSON 导入失败

ACTIVE 创建日期：2026-07-21 所属模块：webserver 需求来源：GitHub Issue #878

1. 原始诉求

Issue 标题：“导入书源 JSON文件过大”；维护者请求：“处理这个 BUG”。

需要修复较大书源 JSON 无法通过管理端导入的问题，并避免破坏电子书分片上传已有的大小约束。

2. 现状与根因

书源粘贴导入会把完整 JSON 作为一次 POST /api/admin/booksource/import 请求发送。开启分片上传后，webserver.main.get_upload_size() 仅使用 UPLOAD_CHUNK_SIZE（默认 4MB）初始化 Tornado 的全局 max_buffer_size。书源导入不属于电子书分片上传，因此超过 4MB 的请求会在到达书源 Handler 前被 HTTP 层拒绝。

书源 JSON（例如 5MB）
        │ 单次 POST
        ▼
Tornado max_buffer_size（当前默认 4MB） ──拒绝──▶ 书源导入 Handler 未执行
        │
        └─修复后粗粒度上限默认 100MB──▶ JSON 解析与书源导入

grill 澄清结果

决策点结论理由
修复层级恢复 HTTP 传输层容量，不改书源 API请求尚未进入 Handler，根因在全局请求缓冲上限
上限算法取 MAX_UPLOAD_SIZE 与 UPLOAD_CHUNK_SIZE 的较大值全局值应容纳所有合法单请求；不能因开启电子书分片而压低其他接口容量
电子书上传约束保持现有 Handler 级分片大小和合并后总大小校验max_buffer_size 是传输层粗粒度保护，不替代业务限制
配置与兼容性不新增配置，不改变 API 和数据结构沿用管理员已有的大小配置即可消除回归
前端改动无前端已经能提交 JSON，失败发生在后端接收阶段

3. 目标与非目标

  • 开启分片上传且分片大小小于最大上传大小时，大于单分片、小于最大上传上限的书源 JSON 可以到达后端接口。
  • 单分片配置大于最大上传配置时，合法的大分片仍可被 Tornado 接收。
  • 不改变书源解析、去重、后台检测和数据库写入逻辑。
  • 不引入书源专用分片协议，也不放宽电子书上传的业务级大小校验。

4. 实现方案

修改 webserver/main.py 的 get_upload_size()：安全解析 MAX_UPLOAD_SIZE 和 UPLOAD_CHUNK_SIZE，无论分片开关状态均返回两者较大值。这样 Tornado 能容纳当前配置允许的最大合法单请求，而具体上传接口继续执行自身限制。

修改 tests/test_main.py 的回归测试，覆盖默认问题组合（最大上传 100MB、分片 4MB）和反向组合（最大上传 100MB、分片 200MB）。

5. 影响范围与风险

范围影响
webserver/main.pyTornado 启动时的全局单请求缓冲上限恢复为两个合法请求规模的较大值
tests/test_main.py锁定全局上限不能低于任一相关配置
API / 数据库 / 前端无变化

风险是全局 HTTP 层可接收比单分片更大的请求；该值仍受管理员配置的 MAX_UPLOAD_SIZE / UPLOAD_CHUNK_SIZE 约束，电子书上传 Handler 的细粒度限制保持不变。
