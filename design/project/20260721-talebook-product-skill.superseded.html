1. 原始诉求

先利用 gh 与 git 分析 Talebook，再制作适合本项目的 SKILL.md。背景目标覆盖产品策划、行业调研和 UI/交互设计，不采用脱离现有产品与工程边界的通用模板。

2. 目标与非目标

目标从 Git 历史、Issue/PR/Release、现有方案和代码结构建立证据链。
目标支持机会识别、竞品研究、优先级判断、交互规格和路线图输出。
目标把自托管、NAS、多用户、跨端阅读和运维成本作为 Talebook 的固有约束。
非目标不默认做商业市场规模测算，不替代实现阶段的仓库方案、测试和浏览器验收。

3. 分析结论

维度仓库证据对 Skill 的要求
产品定位README 将产品定义为基于 Calibre 的个人图书管理系统，强调在线阅读、简单部署、私人模式和多用户。优先服务自托管管理员、个人读者和家庭/小圈子共享，不照搬公有内容平台指标。
核心闭环近一年集中演进导入/扫描、元数据补全、发现与筛选、在线阅读、阅读状态/进度、推送、OPDS、WebDAV。研究和方案必须落入“部署维护 → 入库整理 → 发现组织 → 阅读同步 → 分享推送”闭环。
产品阶段自 v25.06.26 至当前分支约 494 个文件变化；前端迁至 Nuxt 4/Vue 3/Vuetify 3，后端持续扩展服务与安全边界。重点从堆叠能力转向一致性、可发现性、错误恢复、跨端连续性和维护成本。
用户信号Git 历史关联上传重复、导入恢复、PDF/TXT 阅读、翻页、书架、空状态、导航点击、阅读进度等 Issue。不得只按 Issue 数量排序，要聚类到用户任务并验证影响范围。
工程约束现有 AGENTS.md 要求重大改动先方案、配套测试；前端改动还需实际浏览器渲染验收。Skill 的产品产物必须包含边界、状态、异常、权限、响应式与可验收标准。

4. 方案

Skill 放在 .agents/skills/talebook-product/，包含：

  • SKILL.md：触发说明、工作流、证据规则、决策框架、产物规范和实现门禁。
  • references/product-context.md：带日期的产品定位、角色、闭环、现有能力、约束与机会假设。
  • references/research-and-ux.md：竞品分组、研究维度、交互检查表和产物模板。
  • agents/openai.yaml：Codex UI 展示信息与默认调用提示。

5. 设计决策与边界

  1. 优先使用仓库本地证据；需要最新 Issue、PR、Release 或竞品信息时再调用 gh 或浏览器，并标注检索日期。
  2. 把 Issue 当作线索，不把单个请求直接等同于路线图项；先聚类用户任务，再评估受益范围和维护成本。
  3. 竞品分为 Calibre 生态、自托管阅读服务器、漫画/有声等邻近品类和跨端客户端，不做单一功能清单竞赛。
  4. 交互方案必须覆盖加载、空白、失败、权限不足、长任务进度、移动端、深色主题和中英文。
  5. 涉及违法公开传播、绕过付费/反爬或高风险远程执行的方向，明确排除或提高安全门槛。

6. 风险

风险控制
产品快照过期引用资料标注日期，并要求在路线图或行业研究任务中刷新 GitHub 与 Release 证据。
Skill 过宽导致误触发description 限定 Talebook 的产品策划、竞品/行业研究和 UI/UX 任务。
只产出漂亮文档而不可实现强制映射现有页面/API/权限/状态，并给出可验收标准与不做事项。
GitHub 数据不完整记录 CLI 失败，允许用 Git 历史、Release 页面和仓库文件交叉验证，降低证据置信度。
