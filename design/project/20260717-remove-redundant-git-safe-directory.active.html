移除冗余 Git Safe Directory

创建日期：2026-07-17 所属模块：project 状态：ACTIVE 需求来源：PR #873 CI 失败复核
test-server 继续直接使用 talebook/talebook-base:latest 的 Calibre 环境；删除 checkout 后重复执行的 git config --global --add safe.directory。无 Git 时 actions/checkout 使用 REST API 下载源码，后端测试只依赖 base 镜像提供的 make。

原始诉求与证据

目标是让 PR #873 的 CI 通过，同时避免为一条冗余命令向 slim base 引入完整 Git 及 Perl 运行时。失败日志证明 checkout 已先通过 GitHub REST API 完成，随后才在显式 safe-directory 步骤以 git: not found 失败。验收全部 Makefile 测试入口时还发现，testv 与 testvv 仍用已经不存在的 *talebook* 源码路径筛选 coverage，测试本身通过后会因“无数据”退出 1。

证据结论
actions/checkout v4set-safe-directory 默认开启；有 Git 时 action 自行配置，无 Git 时使用 REST fallback，不需要后续 Git 命令。
workflow 历史该步骤曾同时加入 server、UI、integration；UI 与 integration 后来以“冗余”为由删除，server 遗留。
Makefile 动态验证隐藏 Git 后，make init、make lint-py、make pytest 均可解析到原命令并返回成功；Git 变量只供镜像构建标签使用。
执行环境保留 base job container，不在 runner 二次构建 Docker test target。

方案

文件计划改动
.github/workflows/ci.yml删除 Configure Git safe directory 步骤。
tests/test_ci_workflow.py固定 base container、checkout、Makefile 命令及“不再显式执行 safe.directory”的边界。
Makefile将 testv 与 testvv 的 coverage 源码筛选从旧模块名改为当前 webserver/*。
tests/test_makefile.py回归固定 coverage 入口指向当前后端包，避免再次退回旧模块名。

风险与回滚

  • 前置依赖：新版 base 必须包含 make；在镜像发布前，Talebook CI 仍会因缺少 make 失败。
  • Git 能力：test-server 不生成 Git 仓库，不能在后续步骤执行依赖历史的 Git 命令；当前 job 没有此类命令。
  • 回滚：若未来测试确需 Git 历史，应使用明确的 CI/test 镜像或重新设计 checkout，不恢复无依据的冗余步骤。
