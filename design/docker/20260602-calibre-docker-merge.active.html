合并 calibre-docker 基础镜像并改名

统一源码归属，同时保持基础镜像“编译一次、registry 复用”的构建边界

状态 · ACTIVE原日期 · 2026-06-02迁移 · 2026-07-13
TL;DR
将独立仓库 talebook/calibre-docker 的基础镜像源码合并到 Talebook，并把镜像更名为 talebook/talebook-base。基础镜像、构建 workflow 和消费方切换已经落地；原方案中的可覆盖 BASE_IMAGE 构建参数没有实现，主 Dockerfile 仍直接固定为 talebook/talebook-base:8.5。

01原始诉求

原基础镜像由独立仓库维护，包含 Debian 13 Slim、Calibre、Nginx、Supervisor 与 PyQt5。需要统一源码管理并采用更准确的镜像命名，同时不能让主镜像在迁移窗口内失去可拉取的基础镜像。

02目标

  • 基础镜像源码、构建 workflow 与主项目进入同一仓库。
  • 镜像名从 talebook/calibre-docker 改为 talebook/talebook-base。
  • 保留 QEMU 下耗时的 Calibre 构建只执行一次、后续从 registry 复用的机制。
  • 迁移过程不中断主 Dockerfile 与 CI 构建。

03方案

阶段一：建立新基础镜像，不切换消费方

文件改动
Dockerfile.base从原独立仓库迁入，保持内容一致。
.github/workflows/build-base.yml仅在修改基础镜像、推送 base-v* 标签或手动触发时构建和推送。
Makefile增加 build-base 与 push-base。
document/Development.zh_CN.md手动搭建参考链接改为仓库内的 Dockerfile.base。

阶段二：切换消费方

文件当前实现
Dockerfile已切换到新镜像，但使用 FROM talebook/talebook-base:8.5 AS server 直接固定版本；原计划的全局 ARG BASE_IMAGE 与构建参数覆盖未实现。
.github/workflows/ci.yml测试容器已从旧镜像切到 talebook/talebook-base:latest。

版本与触发规则

  • base-v8.5.0 产生 8、8.5、8.5.0 标签；前缀与主项目 v* 发版标签隔离。
  • 修改 Dockerfile.base 后推送 master，产生 latest 与 master。
  • 日常应用代码变更不触发基础镜像重建。

首次引导

make build-base push-base
# 或
git tag base-v8.5.0
git push origin base-v8.5.0
待办
归档原仓库并添加迁移说明；在 Docker Hub 旧镜像页标注迁移；确认历史遗留 Python 包是否可删除；后续单独评估基础镜像中 UID 1000 与主镜像 UID 911 分支的行为。
