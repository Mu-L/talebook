1. 原始诉求

用户确认继续优化现有 talebook-product Skill。优化范围来自上一轮审查：降低纯编码任务误触发，拆分行业调研与交互设计资料，为 Git/GitHub 证据采集增加可重复脚本，把稳定产品上下文与日期快照分离，补充 Talebook 指标库、中文默认提示和正负场景验证。

2. 目标与非目标

触发准确只在产品判断、行业/竞品研究和 UI/UX 设计任务中隐式触发。
上下文经济产品、竞品、UX、指标和历史快照按任务分别加载。
证据稳定用只读脚本统一采集 Git 与 gh 结果，并保留部分失败信息。
可验证验证结构、脚本、正向场景、负向场景和输出契约。

非目标：不改业务代码，不自动发布或操作 GitHub，不把 Skill 拆成多个会互相竞争触发的独立 Skill，不承诺外部网络始终可用。

3. 方案

3.1 触发边界

frontmatter description 明确正向触发：产品分析、Issue 聚类、路线图、竞品研究、PRD、信息架构、用户流程、UI/UX 评审。明确负向边界：纯编码、Bug 诊断、CI、代码审查、部署运维和不涉及产品取舍的 API 实现不隐式触发。

3.2 引用拆分

资源用途加载条件
product-context.md稳定定位、角色、闭环、约束和机会原则所有产品任务
competitor-research.md竞品分组、信源、维度和结论格式行业或竞品任务
ux-design.md页面观察、任务流、状态、可访问性和验收规格UI/UX 任务
product-metrics.mdTalebook 指标候选和选择规则路线图、PRD、实验和复盘
snapshot-20260721.mdV1 分析时点的动态证据与假设需要历史基线时

3.3 证据脚本

新增 scripts/collect_evidence.py，只调用只读的 git 与 gh 子命令；输出 JSON，记录命令状态、标准输出、错误和采集时间。GitHub 查询失败时保留错误但继续输出本地证据，不创建、修改或评论远端对象。

3.4 UX 页面证据

评审现有页面时，只要环境可启动，就必须检查真实渲染页面；记录路由、主题、视口、角色和关键状态。仅做未来概念设计且没有可运行页面时，可以使用流程与线框，但必须明确哪些结论未经真实页面验证。

4. 验证计划

  • 运行 Skill Creator quick_validate.py。
  • 运行证据脚本的 JSON 输出、限制日志数量和 gh 失败降级场景。
  • 检查所有引用均存在、保持一层深度且旧合并引用已删除。
  • 正向场景：版本路线图、跨端竞品研究、批量导入 UX 规格。
  • 负向场景：修复 WebDAV 401、处理 CI 失败、代码审查不应隐式触发。
  • 校验中文 openai.yaml、UTF-8、尾随空白和方案文档。

5. 风险与回滚

风险控制
description 收窄后漏触发保留明确的产品、调研与 UX 同义词，并支持用户显式调用。
脚本输出过大限制日志、Issue、PR、Release 和远端分支数量；允许参数调整。
gh 配置或网络不可用将 gh 作为可降级证据，保留错误而不中断 Git 采集。
引用拆分增加导航成本所有资源从 SKILL.md 一层直达，并写明精确加载条件。

回滚时恢复 V1 Skill 文件和 research-and-ux.md，并将本方案标为 SUPERSEDED。
