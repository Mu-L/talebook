1. 原始诉求与结论

将当前 PR 中新增的 Docker dev stage 从 Node 20 升级到 Node 24。Docker 前端 builder、production-ssr 和 ci.yml 暂不修改，后续通过独立 PR 全面升级。

Node 20 虽能运行当前 Codex 和 Claude Action，但已经停止维护；新建的开发镜像直接采用当前 Active LTS Node 24，避免发布即携带 EOL 运行时，并将支持周期延长到 2028 年 4 月。

本方案仅替代关联基础方案中 dev stage 采用 Node 20 的决定；基础方案的镜像结构、发布策略和其余验证结论继续有效。

2. Node 22 与 Node 24 选型

评估维度Node 22Node 24结论
Node 官方阶段Maintenance LTS，计划 2027 年 4 月 EOLActive LTS，计划 2028 年 4 月 EOLNode 24 多一年维护窗口。
Nuxt 4.3满足 22.12 以上要求满足要求，并符合 Nuxt 推荐 Active LTS 的方向Node 24 更适合新建环境。
Codex 0.144.6声明兼容，实测可安装运行声明兼容，实测可安装运行两者均可；最终镜像保留 codex --version 发布探针。
Claude Code Action v1Action 主入口使用 Bun 并安装原生 Claude CLI，不依赖 dev 镜像预装 Node。Node 选型主要由业务工具链决定。
npm 风险npm 10 路径较保守npm 11 曾有平台可选依赖安装风险用 Codex 启动、前端安装与构建实测控制风险。

以上结论依据 Node.js 发布周期、Nuxt 4 安装要求、Codex 包元数据及 Claude Code Action v1 官方仓库实现，并通过 Node 22 与 Node 24 一次性容器交叉验证。

5. 风险与回滚

  • 风险：Node 24 自带 npm 11，平台可选依赖解析或 Nuxt 工具链行为可能与 Node 20 不同；通过 Codex 启动探针、镜像构建和前端命令验证。
  • 风险：误改其他 stage 会扩大生产影响；契约与 diff 评审限定只出现 dev 安装入口变更。
  • 回滚：将 dev stage 安装入口恢复为 setup_20.x，重新发布浮动 :dev。
