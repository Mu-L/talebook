需求来源
用户反馈：EPUB 转 AZW3 等能力消失，且不可用路线不应显示。

原始诉求与目标

恢复 EPUB 到 AZW3、PDF 的转换能力。转换弹窗只显示当前书籍可立即执行的路线；来源缺失或目标格式已经存在的路线均不显示。全部路线均不可执行时，弹窗只显示明确的“没有可转换的格式”提示。

设计决策

本次以“用户当前能做什么”为唯一展示依据，而不是暴露后端完整的能力清单。弹窗保留书籍格式标签，让空状态仍能解释为什么没有动作可选。视觉上使用书脊绿与琥珀箭头：绿色表达可执行的终点，箭头只用于真实的转换方向，不增加装饰性图表。

路线执行器显示条件
TXT → EPUBtxt2epub-next存在 TXT，且没有 EPUB。
EPUB → AZW3ebook-convert存在 EPUB，且没有 AZW3。
EPUB / AZW3 / MOBI / AZW → PDFebook-convert按优先级选择已有源格式，且没有 PDF；每本书只显示一条 PDF 路线。
AZW3 / MOBI / AZW → EPUBebook-convert当不存在 EPUB 时，按优先级选择一个已有 Kindle 源格式。

流程与边界

后端集中生成完整路线并在其上标记可用性，保留 API 的诊断语义。前端只把 available: true 的项目交给选择器渲染；没有项目时不渲染单选列表，改为显示空状态。接口仍会对提交的来源、目标组合做白名单校验，不能通过伪造参数绕过路线限制。
