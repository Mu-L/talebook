扫描查重作者与入库行为一致性方案

让 do_scan() 的查重判定始终反映 do_import() 实际会做的事，避免“显示可导入、点击却失败”

状态 · ACTIVE创建日期 · 2026-07-16需求来源 · issue #855 / PR #861 review
TL;DR
do_import() 对 PDF/TXT 强制使用“佚名”作为作者写入书库，解析元数据失败时也统一退化为“文件名 + 佚名”。do_scan() 的查重必须使用与 do_import() 完全相同的（标题、作者）组合，否则两者判定不一致：已入库文件在扫描列表中仍显示为“可导入”，点击导入后才因书籍已存在而失败。本方案覆盖三轮修复：查重作者归一化、EXIST 状态落库时机、元数据解析失败兜底一致性。

01原始诉求

图书馆中已有的 PDF/TXT 书籍，在扫描结果中仍显示为“可导入”，实际点击导入会因书籍已存在而失败（issue #855）。PR #861 提交后，Codex 自动 review 又连续发现两个关联问题：EXIST 状态未真正落库、元数据解析失败时的兜底逻辑与 do_import() 不一致。本方案作为这一系列关联修复的统一记录。

02目标

  • 扫描阶段（do_scan()）对同一物理文件计算出的（标题、作者）必须与导入阶段（do_import()）实际写入书库时使用的值一致，查重结果才有意义。
  • EXIST 状态一旦在内存中被判定，必须在返回前完成落库，不能因为提前 continue 被会话回滚吞掉。
  • 元数据解析失败（get_metadata() 抛异常）时，扫描与导入要走同一套“文件名 + 佚名”兜底，且扫描时仍需执行查重，不能跳过。
  • 不改变 do_import() 自身的导入策略，只让 do_scan() 的预测与它对齐。

03现状与三轮修复

轮次问题修复代码位置
1PDF/TXT 查重用文件原始（不可靠）作者比对，而 do_import() 写入时强制用“佚名”，两者永远对不上do_scan() 对 PDF/TXT 也强制 mi.authors = [_("佚名")]，与 do_import() 保持一致webserver/services/scan.py 元数据成功解析分支
2row.status = ScanFile.EXIST 后立即 continue，从未调用 save_or_rollback(row)，变更只留在内存对象上，会话结束即被回滚调整顺序：先无条件 save_or_rollback(row)，再判断是否 EXIST 决定是否 continue同上，查重代码块末尾
3get_metadata() 解析失败时，do_scan() 走独立的 else 分支，用 "Unknown" 作者且完全跳过查重；而 do_import() 的解析失败分支会退化为“文件名 + 佚名”并照常查重——同一份解析失败文件，扫描与导入的“可见状态”不一致解析失败时构造与 do_import() 相同的兜底 Metadata(文件名, [佚名])，复用后续统一的标题/作者归一化与查重逻辑，不再单独短路同上，if not mi: 分支

统一后的处理流程

  1. 读取文件真实哈希，命中已存在真实哈希且对应书籍仍在库时标记 DROP。
  2. 尝试 get_metadata()；成功则使用其返回的 mi，失败则构造 Metadata(文件名, [佚名]) 兜底 —— 两条路径此后共用同一段逻辑。
  3. PDF/TXT 统一将标题改写为文件名、作者改写为“佚名”，与 do_import() 的写入结果对齐。
  4. 调用 books_with_same_title() 查重，标题和作者集合完全一致且格式已存在时标记 EXIST 并记录 book_id。
  5. 无论是否 EXIST，先 save_or_rollback(row) 落库，再决定是否 continue。
已知局限（本轮不修复）
对于书库中已存在、但作者是“真实作者”而非“佚名”的旧版 PDF/TXT（例如直接用 Calibre 导入，或在引入“强制佚名”策略之前导入的书籍），重新扫描同一物理文件时，查重会因为作者不匹配而不标记为 EXIST，扫描列表显示为可导入。这与 do_import() 自身的行为是一致的：do_import() 对 PDF/TXT 永远写入“佚名”，再次导入同一文件本来就不会合并进那本“真实作者”的旧书，而是产生一本新的“佚名”作者书。也就是说，扫描给出的“可导入”提示准确反映了导入后会发生的事，并不构成新的不一致；若要让这类旧书也被判定为重复，需要重新设计整个查重策略（例如引入基于哈希的书库级去重），超出本次缺陷修复范围，留作后续独立方案。
